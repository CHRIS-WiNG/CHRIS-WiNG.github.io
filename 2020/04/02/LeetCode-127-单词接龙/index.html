433. 最小基因变化

Difficulty: 中等

一条基因序列由一个带有8个字符的字符串表示，其中每个字符都属于 "A", "C", "G", "T"中的任意一个。

假设我们要调查一个基因序列的变化。一次基因变化意味着这个基因序列中的一个字符发生了变化。

例如，基因序列由"AACCGGTT" 变化至 "AACCGGTA"即发生了一次基因变化。

与此同时，每一次基因变化的结果，都需要是一个合法的基因串，即该结果属于一个基因库。

现在给定3个参数 — start, end, bank，分别代表起始基因序列，目标基因序列及基因库，请找出能够使起始基因序列变化为目标基因序列所需的最少变化次数。如果无法实现目标变化，请返回 -1。

注意:

  1. 起始基因序列默认是合法的，但是它并不一定会出现在基因库中。
  2. 所有的目标基因序列必须是合法的。
  3. 假定起始基因序列与目标基因序列是不一样的。

Solution

Language: Java

思路1：单向BFS （1512ms）

以单词每次只能变化1个字母为条件，构造出单词变化图，通过每次都找到下一层元素（和当前元素只有一个字母不同的元素）来实现BFS。

思路 2：双端BFS （132ms）

通过单边BFS可以简单实现算法，如果我们还想提高检索效率，可以通过双边BFS来实现。这是因为单边BFS搜索起来就像一颗N叉树，越到底层，队列中元素越多，而正确的endWord只有一个，如果采用双边BFS，可以在一定程度上减少比较的次数；此外，我们还可以每比较完一层计算下一层的时候，选择下一层中元素较少的一端进行遍历，这样可以使算法更具效率 。

思路3：优化比较策略（36ms）

在前两个思路中我们都是从无向图的遍历策略入手，仔细想想其实还可以从单词比较的策略入手，现在的比较策略是：

  • 对每一层需要检测的每一个结点，都去和图中所有的结点进行匹配，把所有匹配到的节点入队，这种策略的时间复杂度和图中的结点数有关，适合wordList中元素个数不多的情况，如果元素个数极多，那这种策略的时间复杂度就很恐怖。
    思考单词特性：因为单词的长度普遍都在十以内，长度不长，并且每个字母只有26种可能，所以换一个角度思考问题：
  • 所以如果我们想探测词典中 一个单词匹配的单词的话，可以去创造出它所有能匹配的单词，并拿这个单词去判断是否是WordList中的单词，这里有一个非常重要的操作！！如果我们去遍历List的话，那么相当于南辕北辙，因为还是需要把List中的单词全部遍历一遍，这里我们创建一个AllWordSet来存放词典中的所有单词，利用哈希的特性来快速检测单词是否存在，相当于快速的把创造出的单词和set取个交集，在基于选择较短边的双端BFS基础上，配合这种策略，效率可以极大提升。
  • 这个思想实际上是把匹配单词的候选集从词典集切换到了 单词创造的单词的集合（可能集），比较这两个候选集，词典集就只和它的容量有关，也就是单词越多，时间复杂度越高；而可能集是和单词长度有关，单词越长，可能集 的时间复杂度越高；而这里由于单词的长度普遍较短，所以使用可能集就会让这道题的时间消耗大大减少。如果这道题的数据是长度比较长的字符串，那使用原始的词典集就要比可能集效率高得多。
